# Systematic Debugging — Detailed Process

> 随机修复浪费时间并制造新 bug。快速补丁只会掩盖深层问题。

**核心原则：** 先找根因，再修复。修复症状是失败。

**沁律：** 未完成第一阶段之前，不得提出修复方案。

---

## 四阶段

必须按顺序完成每个阶段，再进入下一阶段。

### 第一阶段：根因调查

**尝试任何修复之前必须：**

1. **仔细阅读错误信息**
   - 不要跳过错误或警告
   - 它们通常包含精确解决方案
   - 完整阅读堆栈追踪
   - 记住行号、文件路径、错误代码

2. **稳定复现**
   - 能否可靠触发？
   - 精确步骤是什么？
   - 每次都会发生吗？
   - 如果不可复现 → 收集更多数据，不要猜测

3. **检查最近变更**
   - 什么变化可能导致了这个问题？
   - git diff、最近提交
   - 新依赖、配置变更
   - 环境差异

4. **多组件系统中收集证据**

   **当系统包含多个组件（CI → build → signing, API → service → database）：**

   **在提出修复方案之前，添加诊断仪器：**
   ```
   对于每个组件边界：
     - 记录进入组件的数据
     - 记录离开组件的数据
     - 验证环境/配置传递
     - 检查每一层的状态

   运行一次收集证据，显示问题出现在哪里
   然后分析证据确定失败的组件
   最后调查该特定组件
   ```

   **示例（多层系统）：**
   ```bash
   # Layer 1: Workflow
   echo "=== Secrets available in workflow: ==="
   echo "IDENTITY: ${IDENTITY:+SET}${IDENTITY:-UNSET}"

   # Layer 2: Build script
   echo "=== Env vars in build script: ==="
   env | grep IDENTITY || echo "IDENTITY not in environment"

   # Layer 3: Signing script
   echo "=== Keychain state: ==="
   security list-keychains
   security find-identity -v

   # Layer 4: Actual signing
   codesign --sign "$IDENTITY" --verbose=4 "$APP"
   ```

   **这会揭示：**哪一层失败（secrets → workflow ✓, workflow → build ✗）

5. **追踪数据流**
   - 错误值从哪里产生？
   - 谁用错误值调用了它？
   - 一直向上追踪直到找到源头
   - 在源头修复，不是在症状处

   详见同目录下 `root-cause-tracing.md`。

---

### 第二阶段：模式分析

**修复之前先找模式：**

1. **找到正常示例**
   - 在同一代码库中找类似的正常代码
   - 什么正常东西跟出问题的东西相似？

2. **与参考实现对比**
   - 如果是实现某种模式，完整阅读参考实现
   - 不要浏览 — 每行都读
   - 在应用之前充分理解模式

3. **识别差异**
   - 正常和失败之间有什么不同？
   - 列出每个差异，不要假设"那个不重要"

4. **理解依赖**
   - 这个功能需要哪些其他组件？
   - 需要什么设置、配置、环境？
   - 它做出了什么假设？

---

### 第三阶段：假设与验证

**科学方法：**

1. **形成单一假设**
   - 清晰表述："我认为 X 是根本原因，因为 Y"
   - 写下来
   - 要具体，不要模糊

2. **最小化测试**
   - 做最小的变更来验证假设
   - 一次只改一个变量
   - 不要同时修多个东西

3. **继续之前验证**
   - 有效？ → 第四阶段
   - 无效？ → 形成新假设
   - 不要在无效假设上叠加更多修复

4. **不知道时**
   - 说 "我不理解 X"
   - 不要装懂
   - 寻求帮助
   - 继续研究

---

### 第四阶段：实施

**修复根本原因，而不是症状：**

1. **创建失败测试用例**
   - 最简可能的复现
   - 如果可能，写自动化测试
   - 如果没有框架，写一次性脚本
   - 修复之前必须有
   - 使用 TDD skill 写正确的失败测试

2. **实施单一修复**
   - 解决已确认的根本原因
   - 一次只改一处
   - 不要"顺便“改进
   - 不要捆绑重构

3. **验证修复**
   - 现在测试通过了吗？
   - 其他测试还好吗？
   - 问题真的解决了吗？

4. **如果修复无效**
   - 停下
   - 计数：已尝试多少次修复？
   - < 3 次：返回第一阶段，用新信息重新分析
   - **≥ 3 次：停下来质疑架构**
   - 不要在没有架构讨论的情况下尝试第 4 次修复

5. **如果 3+ 次修复失败：质疑架构**

   **表明架构问题的模式：**
   - 每次修复都在不同地方揭露新的共享状态/耦合/问题
   - 修复需要"大规模重构"才能实现
   - 每次修复都在其他地方引发新症状

   **停下来质疑基本原则：**
   - 这个模式根本上是否合理？
   - 我们是否只是"凭惯性坚持"？
   - 我们应该重构架构，还是继续修复症状？

   **在尝试更多修复之前，与人类合作伙伴讨论。**

   这不是假设失败 — 这是架构错误。

---

## 红线 — 停下并执行流程

如果你发现自己在想：
- "先临时修复，以后再调查"
- "试试改 X 看看是否有效"
- "添加多个变更，运行测试"
- "跳过测试，我手动验证"
- "可能是 X，我来修复"
- "我不是完全理解，但这可能有效"
- "文档说 X，但我会以不同方式应用"
- "以下是主要问题：[列出修复而没有调查]"
- 在追踪数据流之前就提出解决方案
- "再尝试一次修复"（已尝试 2+ 次）
- 每次修复都在不同地方揭露新问题

**这些都意味着：停下。返回第一阶段。**

**如果 3+ 次修复失败：** 质疑架构。

## 常见理性化

| 借口 | 现实 |
|--------|---------|
| "问题很简单，不需要流程" | 简单问题也有根本原因。流程对简单 bug 很快。 |
| "紧急，没时间走流程" | 系统化调试比猜测-尝试更快。 |
| "先试试这个，然后再调查" | 第一次修复就建立了模式。一开始就做对。 |
| "确认修复有效后再写测试" | 未经测试的修复不会持久。先写测试才能证明。 |
| "一次修多个地方更省时" | 无法隔离哪个有效。会引发新 bug。 |
| "参考太长，我会适应模式" | 部分理解保证 bug。完整阅读。 |
| "我看到问题了，让我修复" | 看到症状 ≠ 理解根本原因。 |
| "再尝试一次修复" (已失败 2+ 次) | 3+ 失败 = 架构问题。质疑模式，不要再修。 |

---

## 当流程显示"没有根本原因"

如果系统化调查显示问题确实是环境相关、时序相关或外部引起的：

1. 你已经完成了流程
2. 记录你调查了什么
3. 实施适当的处理（重试、超时、错误消息）
4. 添加监控/日志以便未来调查

**但：** 95% 的"没有根本原因"情况都是调查不完整。
